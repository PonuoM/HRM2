# เอกสารความต้องการ

## บทนำ

ฟีเจอร์นี้เป็นการสร้างเว็บแอปพลิเคชันสำหรับการลงเวลาทำงานและการจัดการวันลาของพนักงาน โดยเน้นการใช้งานบนมือถือเป็นหลัก ระบบจะให้พนักงานสามารถบันทึกชั่วโมงการทำงานและขอลาประเภทต่างๆ (ลาพักร้อน ลาป่วย ฯลฯ) ได้อย่างยืดหยุ่นตั้งแต่ 1 ชั่วโมงถึงหลายวัน แอปพลิเคชันจะพัฒนาด้วย PHP 8.0.30 ใช้ฐานข้อมูล MariaDB 10.6.19 และทำงานบนเว็บเซิร์ฟเวอร์ Apache

## ความต้องการ

### ความต้องการที่ 1

**User Story:** ในฐานะพนักงาน ฉันต้องการบันทึกชั่วโมงการทำงานประจำวัน เพื่อให้สามารถติดตามการเข้างานและเวลาทำงานได้อย่างถูกต้อง

#### เกณฑ์การยอมรับ

1. เมื่อพนักงานเข้าถึงหน้าจอลงเวลา ระบบจะต้องแสดงตัวเลือกสำหรับลงเวลาเข้างานและออกงาน
2. เมื่อพนักงานลงเวลาเข้างาน ระบบจะต้องบันทึกเวลาปัจจุบันและรหัสพนักงาน
3. เมื่อพนักงานลงเวลาออกงาน ระบบจะต้องคำนวณและเก็บจำนวนชั่วโมงที่ทำงานในช่วงนั้น
4. เมื่อพนักงานดูประวัติการลงเวลา ระบบจะต้องแสดงรายการเวลาเข้า-ออกทั้งหมดพร้อมชั่วโมงที่คำนวณแล้ว

### ความต้องการที่ 2

**User Story:** ในฐานะพนักงาน ฉันต้องการขอลาประเภทต่างๆ (ลาพักร้อน ลาป่วย ลากิจ) เพื่อให้สามารถจัดการเวลาหยุดพักตามนโยบายบริษัท

#### เกณฑ์การยอมรับ

1. เมื่อพนักงานเข้าถึงฟอร์มขอลา ระบบจะต้องแสดงตัวเลือกประเภทการลาต่างๆ (ลาพักร้อน ลาป่วย ลากิจ ลาฉุกเฉิน)
2. เมื่อพนักงานเลือกประเภทการลา ระบบจะต้องให้ระบุระยะเวลาและวันที่ลา
3. เมื่อพนักงานส่งคำขอลา ระบบจะต้องเก็บคำขอพร้อมสถานะรอการอนุมัติ
4. เมื่อส่งคำขอลาแล้ว ระบบจะต้องแสดงการยืนยันพร้อมรายละเอียดคำขอ

### ความต้องการที่ 3

**User Story:** ในฐานะพนักงาน ฉันต้องการขอลาได้อย่างยืดหยุ่นตั้งแต่ 1 ชั่วโมงถึงหลายวัน เพื่อให้สามารถลาได้ตามความต้องการเฉพาะของฉัน

#### เกณฑ์การยอมรับ

1. เมื่อพนักงานสร้างคำขอลา ระบบจะต้องให้เลือกระยะเวลาเป็นชั่วโมง (ขั้นต่ำ 1 ชั่วโมง) หรือเป็นวันเต็ม
2. เมื่อพนักงานเลือกลาเป็นชั่วโมง ระบบจะต้องให้ระบุเวลาเริ่มต้นและสิ้นสุดภายในวันเดียว
3. เมื่อพนักงานเลือกลาเป็นวัน ระบบจะต้องให้เลือกวันที่เริ่มต้นและวันที่สิ้นสุด
4. เมื่อคำนวณระยะเวลาการลา ระบบจะต้องคำนวณจำนวนชั่วโมงหรือวันที่ขอลาได้อย่างถูกต้อง

### ความต้องการที่ 4

**User Story:** ในฐานะพนักงาน ฉันต้องการดูประวัติการลาและยอดวันลาคงเหลือ เพื่อให้สามารถติดตามการใช้วันลาและวางแผนการขอลาในอนาคต

#### เกณฑ์การยอมรับ

1. เมื่อพนักงานเข้าถึงแดชบอร์ดการลา ระบบจะต้องแสดงยอดวันลาคงเหลือสำหรับแต่ละประเภทการลา
2. เมื่อพนักงานดูประวัติการลา ระบบจะต้องแสดงคำขอลาทั้งหมดในอดีตและที่รออนุมัติพร้อมสถานะ
3. เมื่อแสดงคำขอลา ระบบจะต้องแสดงวันที่ขอ ประเภทการลา ระยะเวลา วันที่ลา และสถานะการอนุมัติ
4. เมื่อการลาได้รับอนุมัติหรือถูกใช้ ระบบจะต้องอัปเดตยอดวันลาคงเหลือของพนักงานโดยอัตโนมัติ

### ความต้องการที่ 5

**User Story:** ในฐานะผู้จัดการ ฉันต้องการตรวจสอบและอนุมัติ/ปฏิเสธคำขอลา เพื่อให้สามารถจัดการความครอบคลุมของทีมและอนุมัติการลาที่เหมาะสม

#### เกณฑ์การยอมรับ

1. เมื่อผู้จัดการเข้าถึงหน้าจออนุมัติ ระบบจะต้องแสดงคำขอลาที่รออนุมัติทั้งหมดของทีม
2. เมื่อผู้จัดการตรวจสอบคำขอลา ระบบจะต้องแสดงรายละเอียดพนักงาน ประเภทการลา ระยะเวลา และวันที่ลา
3. เมื่อผู้จัดการอนุมัติคำขอลา ระบบจะต้องอัปเดตสถานะเป็นอนุมัติแล้วและแจ้งเตือนพนักงาน
4. เมื่อผู้จัดการปฏิเสธคำขอลา ระบบจะต้องอัปเดตสถานะเป็นปฏิเสธและให้เพิ่มเหตุผลการปฏิเสธได้

### ความต้องการที่ 6

**User Story:** ในฐานะผู้ใช้ ฉันต้องการให้แอปพลิเคชันทำงานได้อย่างราบรื่นบนอุปกรณ์มือถือ เพื่อให้สามารถจัดการเวลาและคำขอลาได้ทุกที่ทุกเวลา

#### เกณฑ์การยอมรับ

1. เมื่อผู้ใช้เข้าถึงแอปพลิเคชันบนอุปกรณ์มือถือ ระบบจะต้องแสดงอินเทอร์เฟซที่ตอบสนองและเหมาะสำหรับการสัมผัส
2. เมื่อผู้ใช้นำทางในอินเทอร์เฟซมือถือ ระบบจะต้องให้ปุ่มที่ง่ายต่อการแตะและการนำทางที่ชัดเจน
3. เมื่อผู้ใช้ป้อนข้อมูลบนมือถือ ระบบจะต้องใช้ประเภทการป้อนข้อมูลที่เหมาะสม (ตัวเลือกวันที่ เวลา ดรอปดาวน์)
4. เมื่อแอปพลิเคชันโหลดบนมือถือ ระบบจะต้องรักษาความเร็วในการโหลดและการโต้ตอบที่ราบรื่น

### ความต้องการที่ 7

**User Story:** ในฐานะผู้ดูแลระบบ ฉันต้องการการยืนยันตัวตนและการควบคุมการเข้าถึงตามบทบาท เพื่อให้เฉพาะผู้ใช้ที่ได้รับอนุญาตเท่านั้นที่สามารถเข้าถึงฟีเจอร์ที่เหมาะสม

#### เกณฑ์การยอมรับ

1. เมื่อผู้ใช้พยายามเข้าถึงแอปพลิเคชัน ระบบจะต้องต้องการข้อมูลการเข้าสู่ระบบที่ถูกต้อง
2. เมื่อผู้ใช้เข้าสู่ระบบสำเร็จ ระบบจะต้องสร้างเซสชันที่ปลอดภัยและเปลี่ยนเส้นทางไปยังแดชบอร์ดที่เหมาะสม
3. เมื่อกำหนดสิทธิ์ผู้ใช้ ระบบจะต้องบังคับใช้การเข้าถึงตามบทบาท (พนักงาน vs ผู้จัดการ vs ผู้ดูแลระบบ)
4. เมื่อเซสชันผู้ใช้หมดอายุ ระบบจะต้องเปลี่ยนเส้นทางไปยังหน้าเข้าสู่ระบบและต้องการการยืนยันตัวตนใหม่

### ความต้องการที่ 8

**User Story:** ในฐานะผู้ใช้ ฉันต้องการให้ระบบเก็บข้อมูลอย่างน่าเชื่อถือในฐานข้อมูล MariaDB เพื่อให้ข้อมูลการลงเวลาและการลาของฉันคงอยู่และปลอดภัย

#### เกณฑ์การยอมรับ

1. เมื่อมีการส่งข้อมูลใดๆ ระบบจะต้องตรวจสอบข้อมูลนำเข้าและเก็บไว้อย่างปลอดภัยในฐานข้อมูล MariaDB
2. เมื่อมีการดำเนินการฐานข้อมูล ระบบจะต้องจัดการข้อผิดพลาดอย่างเหมาะสมและให้ข้อมูลป้อนกลับแก่ผู้ใช้
3. เมื่อเก็บข้อมูลที่ละเอียดอ่อน ระบบจะต้องใช้การเข้ารหัสข้อมูลและมาตรการความปลอดภัยที่เหมาะสม
4. เมื่อระบบทำการสืบค้นฐานข้อมูล ระบบจะต้องใช้ prepared statements เพื่อป้องกันการโจมตี SQL injection
